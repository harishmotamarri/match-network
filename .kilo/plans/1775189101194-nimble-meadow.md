# Fix: "Skill not found" Error After Current Availability Step

## Problem

After a user answers the "What's your current availability?" question during profile setup, the app throws `"Skill not found"` and the user sees `❌ Skill not found`.

### Root Cause

The flow works like this:

1. **Profile setup begins** (`bot.handler.ts:137` or `:183`) — `skillsService.getAllSkills()` is called and skill objects (with UUIDs) are stored in the Redis session under `session.tempData.allSkills`.
2. **User selects skills** (`bot.handler.ts:224`) — selected skill objects are stored as `session.tempData.selectedSkills`.
3. **User answers availability** (`bot.handler.ts:266-267`) — the code iterates over `selectedSkills` and calls `skillsService.addUserSkill(userId, skill.id, 3)`.
4. **`addUserSkill`** (`skills.service.ts:20-21`) — re-queries the database with `prisma.skill.findUnique({ where: { id: skillId } })`. If the skill UUID no longer exists in the DB, it throws `'Skill not found'`.

**The stale data problem:** The skill IDs cached in the session can become invalid if:
- The database was re-seeded between steps (UUIDs regenerate because `@default(uuid())`)
- Skills were deleted/recreated in the admin panel
- The Redis `skills:all` cache (24h TTL) served stale data that was already out of sync with the DB

Additionally, there's a **null safety issue**: if `selectedSkills` is `undefined` (e.g., session data was lost), the `for...of` loop at line 266 will throw a TypeError, not "Skill not found" — but both are bugs in the same code path.

## Fix Plan

### 1. Remove redundant DB lookup in `addUserSkill` (Primary Fix)
**File:** `src/modules/skills/skills.service.ts` (lines 19-21)

The `findUnique` check is redundant because:
- The `upsert` on line 23 already includes the `skillId` in the `create` clause, and the foreign key constraint (`skill Skill @relation(...)`) will reject invalid IDs at the DB level.
- The skill objects were already fetched from the DB moments earlier.

**Change:** Remove the `findUnique` + throw guard. Let the `upsert` handle it — if the skillId is invalid, the FK constraint will reject it, and we catch that with a more helpful error message.

```typescript
async addUserSkill(userId: string, skillId: string, proficiencyLevel: number) {
    try {
        const userSkill = await prisma.userSkill.upsert({
            where: { userId_skillId: { userId, skillId } },
            update: { proficiencyLevel },
            create: { userId, skillId, proficiencyLevel },
            include: { skill: true },
        });

        await redis.del(`profile:${userId}`);
        return userSkill;
    } catch (err: any) {
        if (err.code === 'P2003') {
            throw new Error('Skill no longer exists. Please restart profile setup.');
        }
        throw err;
    }
}
```

### 2. Add null safety for `selectedSkills` in `handleProfileAvailability`
**File:** `src/modules/whatsapp/bot.handler.ts` (lines 252, 265-268)

Add a guard to prevent crash if `selectedSkills` is undefined or empty:

```typescript
const { experienceLevel, selectedSkills, city, userName } = session.tempData ?? {};

if (!selectedSkills || selectedSkills.length === 0) {
    // Session data was lost — restart profile setup
    return `⚠️ Your session expired. Let's start your profile setup again.\n\nType *menu* to restart.`;
}
```

### 3. Invalidate Redis skills cache when skills change
**File:** `src/modules/skills/skills.service.ts`

Add cache invalidation in any admin operations that create/update/delete skills. Check if there are admin skill management methods and ensure they call `redis.del('skills:all')`.

### 4. Re-fetch skills at save time instead of relying on session cache (Optional hardening)
**File:** `src/modules/whatsapp/bot.handler.ts` (lines 265-268)

Instead of using `skill.id` from session cache, re-validate by looking up skills by name:

```typescript
// Re-fetch current skills to get valid IDs
const currentSkills = await skillsService.getAllSkills();
const skillNameSet = new Map(currentSkills.map((s: any) => [s.name, s]));

for (const skill of selectedSkills) {
    const current = skillNameSet.get(skill.name);
    if (current) {
        await skillsService.addUserSkill(session.userId!, current.id, 3);
    }
}
```

This ensures that even if UUIDs changed (due to re-seed), the skills are matched by name.

## Summary

The core issue is that `addUserSkill` re-queries the DB for a skill that was already fetched, and the cached skill IDs in the session can go stale. The fix removes the redundant lookup, adds proper error handling, and adds null safety guards for the session data.
